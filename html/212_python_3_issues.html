Issue: Text Encoding (David Beasley)
(35.46 in video)
The internal representation of characters is now almost never the same as how text is transmitted or stored in files
'Hello World' in a text file may just be stored as a series of ascii
bytes such as the following
    list(map(lambda x: hex(x)[2:], list("Hello World".encode())))

    //  ['48', '65', '6c', '6c', '6f', '20', '57', '6f', '72', '6c', '64']
When the file is read into memory, the Python 3 representation
(32-bit integer UCS-4) will be something like the following:
 
    list(map(lambda x: f"000000{hex(x)[2:]}", list("Hello World".encode())))

    //  ['00000048', '00000065', '0000006c', '0000006c', 
         '0000006f', '00000020', '00000057', '0000006f', 
         '00000072', '0000006c', '00000064']
Python 3 will use much more memory!
I/0 Encoding
(David Bleasley, 37.44)
All text is now encoded and decoded
If reading text, it must be decoded from its source format into Python string
If writing text, it must be encoded into some well-known output formula
This it the major difference between Python 2 and Python 3.
In Python 2, you could just write programs that just ignored encoding
and read text as bytes (Ascii)
For binary I/O, Python will never implicitly encode unicode strings and
write them
You must use a text-mode file or explicitly encode str.encode()
